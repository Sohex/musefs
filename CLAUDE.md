# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with
code in this repository.

## What this is

musefs is a **read-only passthrough FUSE filesystem** presenting a re-tagged,
reorganized view of a music library, backed by a SQLite store. The cardinal
invariant: **original audio bytes are never copied or modified** — a served
file is freshly generated metadata spliced in front of positioned reads of
the untouched backing file. Never relax this invariant. The SQLite store is
the source of truth for tags/art and the contract external tools write to.

## The central mechanism (the condensed version)

A synthesized file is a `RegionLayout` (`musefs-format/src/layout.rs`): an
ordered list of `Segment`s — `Inline` generated bytes, `ArtImage`/`BinaryTag`
payloads streamed from the DB at read time, `BackingAudio` positioned reads
of the original file, and `OggAudio`/`OggArtSlice` for Ogg's
patched-in-place pages and incremental base64 art. `read_at`
(`musefs-core/src/reader.rs`) serves any byte range by splicing segments —
audio bytes are never copied. Read
[architecture/serving.md](docs/src/architecture/serving.md#the-segment-model) before touching
read/synthesis paths.

## Code style

musefs is written in clean, performant, idiomatic Rust. The workspace is
strictly layered (db → format → core → fuse/cli/binary); `musefs-core` is
the integration layer — cross-cutting logic belongs there, and `musefs-fuse`
/ `musefs-cli` / the `musefs` binary stay thin.

## Everyday commands

```bash
cargo build                              # build the workspace
cargo test                               # all crates (excludes FUSE e2e)
cargo test -p musefs-core read_at        # one crate / substring filter
cargo test -p musefs-fuse -- --ignored   # FUSE e2e; needs /dev/fuse + libfuse
cargo clippy --all-targets               # lint; benches/ and tests/ only compile here
cargo fmt
```

## Repo-operational facts

- The pre-commit hook runs fmt, clippy (`-D warnings`), the **full workspace
  test suite**, `shellcheck`/`yamllint` over tracked shell/YAML files, and ruff
  over the Python paths — a commit with red tests is always rejected, so TDD
  plans need each commit green. The cargo gate is skipped for docs-only commits
  (every staged path under `docs/` or a `*.md` file); the shell/YAML legs skip
  when their tool is absent.
- The `fuzz/` crate is outside the workspace: format-layer API changes can
  break it silently — `cargo +nightly fuzz build` to check
  ([CONTRIBUTING.md](CONTRIBUTING.md#coverage-guided-fuzzing)).
- After any `musefs-db` schema change, regenerate the Python mirror:
  `MUSEFS_REGEN_SCHEMA_PY=1 cargo test -p musefs-db schema_py`, then
  re-vendor ([CONTRIBUTING.md](CONTRIBUTING.md#python-plugins-contrib)).
- Contrib Python suites have environment gotchas (beets needs a venv;
  real-Picard tests silently skip without an importable Picard) — see
  [CONTRIBUTING.md](CONTRIBUTING.md#python-plugins-contrib) before touching
  `contrib/`.
