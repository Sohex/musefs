# Gating the portable `musefs-fuse` helpers (issue #217)

**Status:** design approved, pending implementation plan
**Issue:** [#217](https://github.com/Sohex/musefs/issues/217) ŌĆö "Mutation-testing
exclusion of musefs-fuse (and parts of musefs-cli) is stale"

## Problem

`.cargo/mutants.toml` excludes `musefs-fuse/**`, `musefs-cli/**`, and the
`musefs` binary from mutation testing wholesale, with a comment justifying it
as "thin glue with no real logic" (per the 2026-05-29 test audit). Since that
audit, `musefs-fuse` accreted a layer of pure, branchy, unit-tested helpers
(largely from the FreeBSD/macOS and FUSE-passthrough work) that are *not* the
thin fuser adapter the comment describes. Those helpers run under the fast
mountless in-diff gate's plain `cargo test`, yet ŌĆö because the whole crate is
excluded ŌĆö their mutants are generated by nothing and verified by nothing. The
exclusion comment is now inaccurate, and genuinely-gateable logic is ungated.

## Investigation (what the mutants actually do)

Rather than reason from the exclusion comment, we ran a real campaign on a
Linux host: `cargo mutants --no-config -p musefs-fuse -p musefs-cli`
(cargo-mutants 27.0.0). Of 116 mutants: **22 caught, 60 missed (survivors),
33 unviable, 0 timeout.** Sorting the survivors/unviable by *cause* overturned
the issue's implicit premise that flipping the exclusion would gate these via
existing tests. The reality is five distinct buckets:

1. **Already gated on the Linux runner (caught, pure, killable):**
   `to_file_attr`, `assemble_dir_listing` (`lib.rs`), and
   `cap_eff_has_sys_admin` (`platform/passthrough.rs`). These are the real win.

2. **macOS-only logic, *uncoverable* on the Linux mutation runner:** the
   Spotlight markers (`platform/spotlight.rs`) and the OS-specific mount path
   (`platform/mount.rs` `extend_os_specific`) exist as `#[cfg(target_os =
   "macos")]` variants alongside `#[cfg(not(macos))]` stubs. On `ubuntu-latest`
   the macOS code is `cfg`'d out, so mutating it changes nothing and its
   mutants **always survive**; its tests (`marker_is_present_on_macos`, the
   macOS mount test) only compile on macOS. Gating this needs a macOS mutation
   leg, which does not exist. The Linux stubs *are* caught (`marker_is_absent_off_macos`).

3. **Unviable (replacement does not typecheck):** e.g. `to_file_attr ->
   Default::default()` (`FileAttr: !Default`), `open_flags ->
   Default::default()`, `parse_mount_config -> (Default, Default)`,
   `From<CliMode> -> Default::default()`. cargo-mutants auto-skips these
   everywhere; they are neither gaps nor concerns.

4. **Equivalent / untestable-by-design:** `reply_errno`'s match-arm only
   selects `log::debug!` vs `log::warn!` ŌĆö the returned errno comes from
   `errno(err)` regardless, so the mutant is behaviorally unobservable (a
   logging-only mutant, like the existing `scan.rs` observability exclusions).
   `definitely_lacks_cap_sys_admin` reads the real `/proc/self/status`, so it is
   not deterministically testable. `MAX_RETAINED_READ_BUF` is a perf/memory knob.

5. **Glue (confirmed survivors, correctly excluded):** every `impl Filesystem
   for MusefsFs` method, `mount`/`mount_with`/`mount_config`/`new_session`/
   `spawn`/`spawn_with`, `fire_poll_refresh`, `notifier_cell` (`lib.rs`);
   `run`/`run_scan`/`run_mount` and the signal handlers `run_unmount`/
   `lazy_detach`/`install_unmount_on_signal` (`musefs-cli`).

### Two findings that shrink the scope

- **`musefs-cli` is effectively a non-issue.** `parse_mount_config` and
  `From<CliMode>` produce *only* unviable mutants ŌĆö zero gateable logic. The
  one viable, caught cli helper is `unmount_commands`, which is not worth
  un-excluding `signal.rs`'s thread/`exit()` glue to reach.
- The valuable, currently-ungated, Linux-killable logic is exactly **three pure
  functions** (bucket 1). They are already caught by existing tests; they are
  simply not *gated* because the crate is excluded wholesale.

## Design

Extract the three genuinely-gateable pure helpers into a single
platform-neutral module so the mutation boundary aligns with a *file* boundary.
This gates them in both the per-PR in-diff gate and the scheduled full
campaign with **zero new `exclude_re` entries**, no cfg-collision handling, and
one small mechanical refactor. Everything else stays excluded ŌĆö but the
exclusion comment is rewritten to be *true*.

### 1. New file: `musefs-fuse/src/convert.rs`

Move, unchanged in behavior, with their tests:

- `to_file_attr` (from `lib.rs`) ŌĆö `Attr` + uid/gid/mtime ŌåÆ `FileAttr`.
- `assemble_dir_listing` (from `lib.rs`) ŌĆö pure readdir-entry assembly.
- `cap_eff_has_sys_admin` (from `platform/passthrough.rs`) ŌĆö pure
  `/proc/self/status` `CapEff` hex-mask parser returning `Option<bool>`. It is
  OS-neutral string parsing; only its *caller* `definitely_lacks_cap_sys_admin`
  touches the OS, and that stays in `passthrough.rs`.

Mechanics:

- `lib.rs` gains `mod convert;` (alongside `mod platform;`). `lib.rs` and
  `passthrough.rs` reference the moved helpers via `crate::convert::ŌĆ”`.
- **Visibility ŌĆö decided per function, not blanket** (a blanket `pub(crate)`
  would break the non-Linux clippy gate; see Risk #2):
  - `to_file_attr` and `assemble_dir_listing` become `pub(crate)`. Both have
    non-`cfg(test)` callers on **every** target (`lib.rs` `getattr`/`lookup`
    and `build_dir_listing` respectively), so they are never dead code.
  - `cap_eff_has_sys_admin` is gated `#[cfg(any(target_os = "linux", test))]`
    and kept `pub(crate)`. Its only non-test caller
    (`definitely_lacks_cap_sys_admin`) is `#[cfg(target_os = "linux")]`, so on
    macOS/FreeBSD a `pub(crate)` fn compiled into the **non-test** lib build
    would be unused ŌåÆ `dead_code` ŌåÆ fails the `-D warnings` gate (the macOS job
    is the project's only non-Linux clippy gate; FreeBSD is cross-linted). The
    `cfg(any(linux, test))` gate makes it compile exactly where it is used: the
    Linux lib build (caller present) and every platform's **test** build (the
    moved tests reference it). `pub` (unconditionally reachable, no warning)
    is the simpler alternative but leaks an internal `/proc` parser into the
    crate's surface and breaks the codebase's cfg-discipline; prefer the
    `cfg`-gated form. The Linux mutation runner compiles it either way, so
    gating is unaffected.
- **Tests are *split out*, not moved wholesale.** `lib.rs`'s single
  `#[cfg(test)] mod tests` mixes the two relocating tests
  (`converts_dir_and_file_attrs`, `assemble_dir_listing_puts_dot_and_dotdot_first`)
  with tests that **stay** (`fuse_config_default_is_conservative`,
  `open_flags_*`, the `poll_pending`/`fire_poll_refresh` tests); `mod
  errno_tests` also stays. Extract only those two named tests into
  `convert.rs`'s test module and replace their `use super::*` with explicit
  imports (`FileType`, `Attr`, `Duration`, `SystemTime`, `INodeNo`, etc.) ŌĆö do
  not move the whole module. The four `cap_eff_parser_*` tests move from
  `passthrough.rs`'s `#[cfg(target_os = "linux")] mod imp::tests` into
  `convert.rs`; being OS-neutral they now run on **every** platform (a coverage
  improvement) and exercise the `cfg(any(linux, test))` parser in test builds.
- `definitely_lacks_cap_sys_admin` in `passthrough.rs` changes its call site
  from `cap_eff_has_sys_admin(...)` to `crate::convert::cap_eff_has_sys_admin(...)`.

**Stays in `lib.rs` (excluded glue), explicitly *not* moved:** `errno`,
`open_flags` (only-unviable mutants ŌåÆ nothing to gate), `reply_errno`
(equivalent logging mutant), `build_dir_listing` (takes `&Musefs`, core I/O ŌĆö
see "Out of scope"), `mount_config`, the `Filesystem` impl, session functions,
`MAX_RETAINED_READ_BUF`.

After this move, `convert.rs` is expected to be **survivor-free**: every viable
mutant in `to_file_attr`, `assemble_dir_listing`, and `cap_eff_has_sys_admin`
was caught at the helpers' *current* locations in the investigation run, and the
mutant *shapes* are identical at the new path. cargo-mutants regenerates the
mutant set for the relocated `convert::cap_eff_has_sys_admin` (the investigation
saw it as `imp::cap_eff_has_sys_admin`), so the survivor-free property must be
**re-verified at the new path** ŌĆö this is exactly the ┬¦"Success criteria" check.

### 2. `.cargo/mutants.toml`

Narrow the fuse glob so only `convert.rs` is in scope; rewrite the comment.

```toml
exclude_globs = [
    "musefs-fuse/src/lib.rs",
    "musefs-fuse/src/platform/**",
    "musefs-cli/**",
    "musefs/**",
    "musefs-core/src/metrics.rs",
]
```

The replacement comment must state the accurate rationale (replacing "thin
glue with no real logic"):

- `musefs-fuse/src/convert.rs` is **in scope** ŌĆö the portable pure helpers
  (`to_file_attr`, `assemble_dir_listing`, `cap_eff_has_sys_admin`), gated in
  both the in-diff gate and the full campaign.
- `musefs-fuse/src/lib.rs` is excluded: the `Filesystem` trait impl + session
  glue (all-survivors), plus helpers whose only mutants are unviable
  (`errno`, `open_flags`), equivalent (`reply_errno` logging), or core-I/O
  (`build_dir_listing`).
- `musefs-fuse/src/platform/**` is excluded: the logic-bearing parts are
  `#[cfg(macos)]` and uncoverable on the Linux mutation runner (mutants always
  survive there; tests only run on macOS). Gating them requires a macOS
  mutation leg ŌĆö deferred. `cap_eff_has_sys_admin` was the one OS-neutral
  exception and has moved to `convert.rs`.
- `musefs-cli/**` is excluded: `parse_mount_config`/`From<CliMode>` produce
  only unviable mutants; `unmount_commands` is the lone caught helper and is
  not worth un-excluding `signal.rs`'s thread/`exit()` glue for.
- `musefs/**` (the binary) remains correctly excluded (one-line entrypoint).

No `exclude_re` entries are added, so `scripts/check_mutant_anchors.py` is
unaffected (it governs only `exclude_re`). The guard's "an `exclude_re` match
must not land in an `exclude_globs` file" invariant still holds, because no
existing `exclude_re` targets `musefs-fuse` (the sole `convert\.rs` anchor is
crate-prefixed `musefs-format/src/convert\.rs` and cannot match the new fuse file).

**Glob-fragility note (behavior change to document in the comment):** narrowing
from `musefs-fuse/**` to `lib.rs` + `platform/**` means any *future* top-level
file added under `musefs-fuse/src/` (a sibling of `convert.rs`) is **silently
in scope** for mutation. That is the intended default (new logic should be
gated), but it is a change from "whole crate excluded" ŌĆö note it so a future
contributor isn't surprised when a new `musefs-fuse/src/foo.rs` starts producing
gate failures.

### 3. `scripts/mutants.sh`

Add a `musefs-fuse` leg to the per-crate dispatch, scoped to the new file:

```sh
musefs-fuse)
  run_crate musefs-fuse --test-workspace=false \
    --file musefs-fuse/src/convert.rs
  ;;
```

This leg runs the default `cargo test` (no `--ignored`, no mounted tests), so
it is **mountless** ŌĆö unlike the `musefs-latencyfs` leg it needs no `/dev/fuse`
at runtime. Update the script's header comment block (`scripts/mutants.sh:5-6`),
which currently asserts "musefs-cli, musefs-fuse, and the `musefs` binary are
thin glue with no real logic and are excluded from mutation entirely" ŌĆö rewrite
it so it no longer claims `musefs-fuse` is excluded entirely: state that
`musefs-fuse/src/convert.rs` is mutation-tested (this leg) while the trait
adapter, session glue, and `cfg(macos)` platform code remain excluded, and that
`musefs-cli` + the `musefs` binary stay fully excluded. The crate is **not**
added to the default crate list (`musefs-db musefs-core musefs-format`); it runs
when named explicitly and from the CI matrix. The fuse leg is intentionally
**unsharded** (no `shard` key in the matrix row ŌåÆ `MUTANTS_SHARD` empty ŌåÆ
whole-crate; `convert.rs` is tiny, so sharding it buys nothing).

### 4. `.github/workflows/mutants.yml`

- **`full` matrix:** add `- { crate: musefs-fuse, label: musefs-fuse }`. No
  libfuse install step is gated on it (the leg never mounts; it only needs
  whatever the normal `musefs-fuse` build needs, which the existing CI build
  already satisfies ŌĆö see Risks). **This row and the ┬¦3 `mutants.sh` `case` arm
  are coupled and MUST ship in the same commit:** the matrix leg invokes
  `scripts/mutants.sh musefs-fuse`, which exits with "unknown crate" if the ┬¦3
  arm isn't present ŌĆö a partial landing yields a red scheduled job.
- **`in-diff` gate:** **no change required.** `convert.rs` is automatically
  in scope once removed from `exclude_globs`; `lib.rs` and `platform/**` stay
  excluded via the narrowed globs; the existing `--exclude
  'musefs-latencyfs/**'` is untouched. A PR that changes `convert.rs` lines is
  now gated; a PR that changes `lib.rs`/`platform/**` glue is not (honest, no
  false survivors).
- The workflow already triggers on `.cargo/mutants.toml` changes, so this PR's
  config edit runs the anchor guard and the in-diff gate.

## Out of scope (documented, deliberately deferred)

- **`build_dir_listing` gating.** It is `fn(&Musefs, ino) -> ŌĆ”` doing core I/O
  (`core.readdir`, `core.parent`) over the pure, already-gated
  `assemble_dir_listing`. Its `-> Ok(vec![])` survivor is killable only by a
  fuse-level test with a live `Musefs` (temp `Db` + `Musefs::open` + inserted
  track). Since every constituent is already gated (readdir/parent in
  `musefs-core`, assembly in `convert.rs`) and the wiring is exercised
  end-to-end by the `#[ignore]d` e2e `readdir` tests, a dedicated fixture test
  is low-yield. Left in `lib.rs` (excluded) and documented.
- **macOS Spotlight / OS-specific mount mutation.** Requires a macOS mutation
  leg in CI (the macOS runner currently does clippy/tests only). Deferred; the
  comment records why.
- **`musefs-cli` mutation.** No viable logic mutants worth the glue noise.

  (`cargo test`), and the four `cap_eff_parser_*` tests now run on all platforms.
- Re-run the scoped campaign locally to confirm `convert.rs` is survivor-free:
  `cargo mutants -p musefs-fuse --file musefs-fuse/src/convert.rs` (mountless;
  use a `/tmp` `TMPDIR` per the local-mutants convention). Expect 0 missed.
- Run the anchor guard locally against the edited toml:
  `cargo mutants --no-config --list --json > /tmp/m.json && python3
  scripts/check_mutant_anchors.py --mutants-json /tmp/m.json` ŌĆö it must still

- `cargo fmt --all --check` and `cargo clippy --all-targets` before pushing.
- **Non-Linux cross-lint** (the `cap_eff_has_sys_admin` move is the reason):
  `cargo clippy --target x86_64-unknown-freebsd --all-targets` must be clean ŌĆö
  specifically no `dead_code` on the relocated cap parser (see Risk #2).
- The `fuzz/` crate is unaffected (no format-layer API change).

## Risks & open items for the plan

- **libfuse at build time (build check, low risk).** The campaign leg builds
  `musefs-fuse`'s test binary. The in-diff gate builds the whole workspace today
  with **no** fuse apt step, and `musefs-fuse` is a workspace member, so it
  already links without `libfuse3-dev`; the `full` leg adds no mount. Treat this
  as a one-line plan check ("confirm the `full` musefs-fuse leg builds without
  the fuse3 install step"), not an open design risk.
- **Dead-code on non-Linux ŌĆö the load-bearing constraint, resolved in ┬¦1.**
  `cap_eff_has_sys_admin` compiles on all targets but its only non-test caller
  is `cfg(linux)`. A naive `pub(crate)` would be **dead in the non-test lib
  build** on macOS/FreeBSD (rustc only treats `pub` items as
  always-reachable; a `pub(crate)` item used solely from `#[cfg(test)]` code is
  *not* reachable for the lint) ŌåÆ `dead_code` ŌåÆ fails the `-D warnings` clippy
  gate (macOS is the only non-Linux clippy gate; FreeBSD is cross-linted). The
  ┬¦1 mechanics resolve this by gating the fn `#[cfg(any(target_os = "linux",
  test))]`. The plan must implement that gate (or unconditional `pub`) and
  **must not** fall back to a blanket `pub(crate)`. Verify with a non-Linux
  cross-lint (`cargo clippy --target x86_64-unknown-freebsd --all-targets`,
  per the FreeBSD lint convention) that no `dead_code` fires.
- **No regression in the macOS/non-Linux path.** Moving `cap_eff_has_sys_admin`
  out of the `cfg(linux) mod imp` must not change behavior on Linux, and the
  `cfg(not(linux))` stub never referenced it, so the stub is untouched. Low
  risk; the cross-lint above is the check.

## Success criteria

- `musefs-fuse/src/convert.rs` exists with the three pure helpers + tests; the
  whole workspace builds and tests green (pre-commit clean).
- `cargo mutants -p musefs-fuse --file musefs-fuse/src/convert.rs` reports zero
  survivors.
- `.cargo/mutants.toml`, `scripts/mutants.sh`, and `mutants.yml` reflect the new
  scope, and the exclusion comments are accurate (no "no real logic" claim
  about a crate that has some).
- The anchor guard passes; no new `exclude_re` introduced.
